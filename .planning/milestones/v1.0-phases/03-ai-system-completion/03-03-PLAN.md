    **ai/approval.ts — resolveApproval erweitern:**

    Den bestehenden `resolveApproval` Code so aendern, dass bei `requests.get(requestId) === undefined` ein DB-Lookup versucht wird:

    ```typescript
    async resolveApproval(requestId, status, reason?) {
      let request = requests.get(requestId);

      // Fallback: DB-Lookup wenn nicht in Memory (z.B. nach Server-Restart)
      if (!request && deps.loadRequest) {
        const dbRequest = await deps.loadRequest(requestId);
        if (dbRequest) {
          request = dbRequest;
          requests.set(requestId, request); // In Memory cachen
        }
      }

      if (!request) {
        throw new Error(`Approval request "${requestId}" not found`);
      }

      if (request.status !== "pending") {
        throw new Error(`Approval request "${requestId}" is already resolved (${request.status})`);
      }

      // ... bestehende Logik (In-Memory update + DB update)
    }
    ```

    Dies loest den Hybrid-Ansatz-Blocker: Wenn die In-Memory Map leer ist (nach Restart), wird der Request aus der DB geladen. Der Agent haengt nicht, weil `resolveApproval` jetzt auch ohne Memory-State funktioniert.

    **WICHTIG: Die waitForApproval()-Problematik:**
    `ai/approval.ts` hat aktuell KEIN `waitForApproval`. Das ist in `services/approval.ts`. Der Agent-Code nutzt `requestApproval()` und dann wartet der Agent-Loop via Vercel AI SDK's tool-call-Mechanismus — NICHT via explizites Promise-Warten. Der `resolveApproval` wird ueber die REST-Route aufgerufen. Daher ist ein Promise-basiertes `waitForApproval` hier nicht noetig.

    Falls in Zukunft `waitForApproval` gebraucht wird (z.B. fuer synchrone Agent-Ausfuehrung), kann es als optionale Erweiterung hinzugefuegt werden. Fuer Phase 3 ist der REST-basierte Resolve-Flow ausreichend.
