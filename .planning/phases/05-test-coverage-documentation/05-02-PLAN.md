    Szenarien:
    - Test 1: Gueltiger Token — Request mit "Bearer valid-token" → verifyToken gibt User zurueck → HTTP 200, Response-Body enthaelt Daten
    - Test 2: Abgelaufener Token — Request mit "Bearer expired-token" → verifyToken gibt null zurueck → HTTP 401
    - Test 3: Cache Hit — Zwei Requests mit demselben Token → verifyToken wird nur 1x aufgerufen (zweiter Request nutzt Cache)
    - Test 4: Cache Miss — Zwei Requests mit verschiedenen Tokens → verifyToken wird 2x aufgerufen
    - Test 5: Malformed Token — Request mit "InvalidFormat" (kein "Bearer " Prefix) → HTTP 401
    - Test 6: Kein Token — Request ohne Authorization Header → HTTP 401
    - Test 7: Permission Denied — Gueltiger Token aber hasPermission gibt false zurueck → HTTP 403
  </behavior>
  <action>
    Per D-06: Erstelle `template/backend/src/auth/auth-flow.integration.test.ts`.

    Setup: Hono-App mit createModuleAuthMiddleware und geschuetzter Route. Nutze `app.request()` fuer HTTP-Tests (kein laufender Server noetig, per D-01).

    Die Integration unterscheidet sich von den bestehenden Unit-Tests in module-auth-middleware.test.ts dadurch, dass der Cache-Mechanismus getestet wird. Dafuer braucht der verifyToken-Mock einen State (Map), der Calls trackt:

    ```typescript
    import { describe, it, expect, mock, beforeEach } from "bun:test";
    import { Hono } from "hono";
    import { createModuleAuthMiddleware, type ModuleAuthMiddlewareDeps } from "./module-auth-middleware";
    ```

    Cache-Simulation:
    - Erstelle einen `verifyToken`-Mock der intern eine Map nutzt um zu tracken welche Tokens schon verifiziert wurden
    - Beim ersten Aufruf mit einem Token: Echte Logik ausfuehren, Ergebnis in Map speichern
    - Beim zweiten Aufruf mit demselben Token: Ergebnis aus Map zurueckgeben
    - Zaehle die tatsaechlichen "Netzwerk-Calls" (= Aufrufe die NICHT aus dem Cache kommen)

    WICHTIG: Die bestehende module-auth-middleware.ts hat KEINEN eingebauten Cache. Der Cache-Test verifiziert, ob die Middleware den verifyToken-Aufruf fuer identische Tokens innerhalb einer Session wiederholt oder nicht.     Pruefe im `read_first`-Schritt ob die Middleware einen Cache hat. Falls NEIN:
    - Test 3 (Cache Hit): Erwartung anpassen → verifyToken wird 2x aufgerufen (kein Cache)
    - Test 4 (Cache Miss): Wird zu "verschiedene Tokens werden korrekt unterschieden"
    - Einen Kommentar im Test hinterlassen: "// Middleware hat keinen Token-Cache — jeder Request verifiziert neu"
